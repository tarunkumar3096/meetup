Input Subsystem

  • Part of Linux Kernel.
  • Manages various Input devices.
  • Included in the kernel because
    • Devices usually accessed through special hardware interfaces.
    • which are protected and managed by the kernel.

Elements in Input Subsystem

  1. Input core
  2. Drivers
  3. Event handlers
    • Various Elements Communicate through EVENTS

Working

  1. Drivers Communicate with low-level hardware.
  2. Drivers convert hardware version data into Standard event format.
  3. Event is sent to the input core.
  4. Input core uses standard kernel plugin design with
    1. input_register_device() - to add a device.
    2. input_unregister_device() - to remove a device.
      1. both uses input_dev structure as argument.
      2. input_dev structure holds specific capabilities of a device, type of input, device type, etc..
  5. To make use of input_events from core, handlers provides user space interface.

Events

type

  • generic type of event being reported.
  • Eg:
    • a key press (EV_KEY)
    • relative motion (like moving a mouse) (EV_REL)
    • absolute motion (like moving a joystick) (EV_ABS)

code

  • tells which of the various buttons or axes are being manipulated
  • Eg:
    • KEY_S, KEY_CAPSLOCK, KEY_ESC, KEY_SPACE
    • REL_X, REL_Y
    • BTN_JOYSTICK, BTN_TRIGGER

value

  • tells what the state or motion is.
  • Eg:
    • 1 for pressed, 0 for released

Application Interface:

uevent:

uinput:

  • A kernel module.
  • Emulate input devices from userspace.
    • Creates Virtual Input Device.
    • Events are passed through it.
    • Delivered to userspace.

Steps to access uinput:

  1. Open File.
  2. Set event.
  3. Register keys.
  4. Set Device Info.
  5. Create virtual device.
  6. Write data to device.
  7. Destroy virtual device.

Interface:

linux/uinput.h
  • Defines ioctl's request to create, set up, and destroy virtual devices.
